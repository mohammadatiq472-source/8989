## 1. 我们当时定义的项目是什么

我们当时定义的是一个 **AI 原生同盟战争 / SLG 指挥系统**：

1. 真人玩家扮演盟主、指挥官、管理层。
2. AI 扮演盟员、官员、执行者。
3. 真人负责战略目标与优先级。
4. AI 负责任务拆解、执行、复盘与建议。

当时强调“不是外挂式 AI 代点”，而是原生可结构化理解的世界系统。

---

## 2. 当时明确的要解决问题

历史提示词中聚焦的核心痛点：

1. 大量真人协作成本高。
2. 管理层长期指挥疲劳。
3. 普通玩家难体验“指挥快感”。
4. 执行力依赖大量真人在线和重复劳动。
5. 闭源手游外挂路线（截图/OCR/模拟点击）成本高、风险高、不可持续。

因此历史方向是：直接做“天生适合 AI 参与”的原生系统。

---

## 3. 当时定义的核心特色（保留价值高）

1. 真人定战略，AI 组织执行。
2. 状态/资源/目标/行动尽量结构化。
3. API 化/JSON 化/工具化。
4. AI 通过结构化状态行动，而非看图乱点。
5. 乐趣核心是管理感、指挥感、组织感。
6. 目标形成“少量真人高层 + 大量 AI 盟员”的组织体验。

---

## 4. 历史架构主轴（原始意图）

历史分层定义如下：

1. 人类玩家（战略意图）
2. CommanderAgent（战区级计划）
3. GeneralAgent（战线/职能执行）
4. UnitAgent 或规则层（低成本大量执行）
5. 规则引擎（authoritative 裁决）

历史核心原则：

1. AI 只能提案，不能直接改世界。
2. 规则引擎永远是最终裁决者。
3. 玩家主要塑造 Doctrine，不做逐条微操审批。
4. 决策过程要可追溯、可复盘、可叙事化。

## 7. 历史 AI 设计哲学（仍有部分长期有效）

历史文档强调：

1. AI-First API Design（语义化字段、减少歧义）
2. 低幻觉原则（结构化输入、Zod 守卫、缺情报先侦察）
3. 成本金字塔（Commander 强模型、General 中等、Unit 低成本）

其中“AI-First、低幻觉、规则引擎裁决”仍然有效。

---

## 8. 历史非目标（多数仍有效）

历史文档中非目标包括：

1. 不做完整 MMO 正式联机（当时阶段）
2. 不做大规模自治 swarm 表演
3. 不把规则逻辑塞进 prompt 替代规则引擎
4. 不在前端保留 authoritative 分支

---

## 9. 历史自主性分级（持续有效）

历史定义三级：

1. L1 `assigned`：玩家在线，AI执行并反馈风险
2. L2 `delegated`：玩家离线或授权托管，AI完整决策
3. L3 `negotiated`：跨势力 AI 协商（结盟/停火/博弈）

该分级是当前 session/autonomy 体系的重要来源。
