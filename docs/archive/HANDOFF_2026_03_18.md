## 一、本次会话完成了什么（最重要）

### 1.1 BUG 修复：战斗触发归零 → 8 场战斗/20回合

**症状**：20 回合内 `battleCount = 0`，所有势力单位根本不向洛阳行军。  
**根因链（完整追溯）**：

```
mockPlanner 产出 6 orders（3个 luoyang_advance + 3个 pve_advance）
    ↓
createCommanderPlan → guardPlan（守卫层）
    ↓
guardPlan.allowedTiles = worldSnapshot.localLayer.tiles（仅约 20 格半径内）
    ↓
洛阳 tile ID = 'luoyang_4_4'（格式：前缀_localX_localY）
    ↓
'luoyang_4_4' NOT IN localLayer → guard_target_missing → 3个洛阳行军指令DROPPED
    ↓
只剩 3 个 pve_advance orders → 单位永远在家附近扩张
    ↓
battleCount = 0（各势力永不碰面）
```

**修复位置**: `server/src/agents/commander/CommanderAgent.ts` `guardPlan` 函数

```typescript
// 修复前（错误）:
if (!allowedTiles.has(order.target)) {    // 洛阳格子不在 localLayer → 直接丢弃
  reasons.add('guard_target_missing')
  continue
}

// 修复后（正确）:
const globalTilePattern = /^[a-z]+_\d+_\d+$/
const targetValid = order.action === 'march'
  ? (allowedTiles.has(order.target) || globalTilePattern.test(order.target))
  : allowedTiles.has(order.target)   // capture/garrison 等仍限 localLayer
```

**验证结果**：
- T3 时 `han_1` 和 `han_5` 已抵达洛阳坐标 `(160, 158)` ✓
- 20 回合仿真：`battleCount = 8`（从 0 → 8）✓

## 三、实测发现：哪些 AI 功能没被充分利用

### 3.1 完全没被利用到的功能

**① PlanningJobMachine（XState 状态机）**
- **问题**：`runMultiFactionSimulation.ts` 主循环直接调用 `createCommanderPlan()`，完全绕过了 `PlanningService`，XState 状态机（queued→running→succeeded/failed/stale）在仿真中从未被触发
- **价值**：状态机是用来追踪"正在规划中/规划失败/计划过期"等生命周期的，对多势力并发规划非常重要
- **建议**：把仿真主循环改为通过 `PlanningService.createPlanAndQueue()` 路径，XState 自然接入，不需要额外写代码

**② GeneralNegotiationChannel（将领谈判通道）**
- **问题**：`GeneralNegotiationChannel.ts` 已完整实现（距离≤5格触发谈判消息），但在仿真 tick 中从未调用
- **价值**：这是"AI 与 AI 之间真实社会层"的第一个具体实现——将领在战场相遇时博弈
- **建议**：在 `runMultiFactionSimulation.ts` 的 GeneralDispatch 阶段后，对所有距离≤5格的跨势力将领对，触发一次 `negotiationChannel.exchange()`

**③ DiplomacyAgent 的外交协议实际效力**
- **问题**：`ceasefire/alliance` 协议写入 `world.feedback.diplomacyAgreements`，但规则引擎 `advanceTick` 在处理进攻命令时**不检查停战协议**，外交完全是 NPC 对话而非真实约束
- **价值**：如果外交协议真的能约束进攻，"背叛"事件就有了真实游戏价值；将领背叛事件写入叙事流和记忆层也会更有意义
- **建议**：在 `executeOrderStep` 处理 `march/capture` 时，检查 `world.feedback.diplomacyAgreements` 中是否有与目标势力的有效停战协议，若有则阻止或触发背叛判定

**④ Voyager 模式技能提炼（addTacticalSkill）**
- **问题**：`ReflectService.ts` 中有完整的 `addTacticalSkill` 实现（Voyager 风格技能自我提炼），但 skill 被写入 GeneralProfile 后，`GeneralUtilityAI` 在决策时**没有读取 skills 字段**
- **价值**：将领通过战斗积累技能，然后自主使用这些技能改变决策——这正是"AI 进化型将领"的核心玩法
- **建议**：在 `GeneralUtilityAI.evaluateGeneralOptions()` 中，读取 `general.memory.skills`，对特定战术选项给予加权

### 3.2 实现了但有缺陷的功能

**① Mem0 记忆层 — commanderId 未按势力隔离**

- **问题**：`resolveCommanderAgentId(factionId)` 只区分 `'player'` 和 `'enemy'` 两组，13势力仿真时所有非 `han` 势力的 Commander 都共享 `commander_player` 的记忆空间
- **影响**：记忆会在13个势力之间相互污染，"魏国Commander记住了吴国上回的战败"完全没有意义
- **代码位置**：`CommanderAgent.ts` 的 `resolveCommanderAgentId()` 函数
- **修复方案**（简单）：
  ```typescript
  function resolveCommanderAgentId(factionId: string): string {
    return `commander_${factionId}`  // 每个势力独立记忆空间
  }
  ```

**② MCP Server — 独立进程架构问题**

- **问题**：`server/src/mcp/gameServer.ts` 直接 import `WorldService`，在 `.vscode/mcp.json` 的 stdio 启动模式下，MCP Server 是独立 Node.js 进程，与游戏后端进程的 WorldService 不共享内存，`get_world_summary` 等工具操作的是空 WorldState（所有 tick=0）
- **影响**：MCP Tools 在 VS Code 中能调用但返回的都是空数据或初始状态
- **修复方案**：让 MCP Server 通过 HTTP 调用 `http://localhost:8787/api/...` 而不是直接 import，或者在 `server/src/app.ts` 中同进程启动 MCP 服务

**③ `han` 和 `yi`（益州）共享 anchorId**

- **问题**：`runMultiFactionSimulation.ts` 中 `han`（汉中）和 `yi`（益州）都配置了 `anchorId: 'yizhou'`
- **影响**：若有基于 anchorId 的查找逻辑（如 `getProvinceMeta`）会产生歧义；实际上 BFS 扩张因坐标不同仍正常（han 起始 (104,200)，yi 起始 (80,246)），但省级元数据层有混淆
- **修复**：将 `han` 的 anchorId 改为 `'hanzhong'` 并在 scenario.ts 中添加对应 anchor

**④ getFullPath 贪心算法在真实地形下不可靠**

- **问题**：当前使用曼哈顿贪心 BFS，在有 U 形山地障碍时会陷入死角，返回空路径
- **影响**：实际仿真中通过"清除地形障碍"规避了此问题，但正式游戏地形会暴露
- **修复方案**：换用 A*（预期开销：3-5小时，`shared/domain/rules.ts` 约 2200 行附近）

### 3.3 实验发现可以精简或移除的功能

**① processAutoExpansion 中的 BFS 自动扩张**

- **发现**：20 回合内，BFS 自动扩张比手动 march/capture 命令更占主导。"自动扩张"逻辑让 AI 计划里的 `expand` 动作变得多余——即使 Commander 不下任何命令，势力领土也在被动增长
- **评估**：对于"体现 AI 战略智能"的目标而言，这个机制是干扰项——它模糊了"AI主动决策"和"规则引擎被动扩张"之间的边界
- **建议**：保留 BFS 扩张作为领土扩散的物理基础，但限制其配额（单位数×5格而非×30格），让 AI 的显式 march/capture 指令成为主要扩张手段

**② processAutoLevelUp（每tick1队+5级）**

- **发现**：每 tick 全势力自动升级，导致实际战斗力差异在 5 tick 后趋于均等（所有势力武将都接近满级）
- **评估**：这使得"弱小势力被初期强势力消灭"的自然淘汰机制缺失——20 回合后 13 个势力均存活，无一被灭
- **建议**：将 AutoLevelUp 改为经验点机制（战斗获得经验→升级），而不是被动时间升级

**③ CopilotKit Sidebar 对于策略玩家体验的定位模糊**

- **发现**：CopilotSidebar（侧边栏聊天）默认开启。但在实际游戏流程中，玩家更自然的交互是"点击将领→下指令"，而不是"打开聊天框→对话"
- **评估**：CopilotKit 的价值更适合放在"将领反馈流"（将领主动向玩家报告）而不是"玩家主动聊天"
- **建议**：将 CopilotSidebar 改为折叠默认关闭，主要展示 `useCopilotReadable` 推送的世界快照和叙事事件流；"下命令"改走 `GeneralChatPanel` 的专属命令模式

**④ GeneralAgent 完整 LLM 调用（gateway模式）在13势力规模下成本不可行**

- **发现**：gateway 模式下，13势力×每势力3-5将领×1000ms LLM延迟 = 每tick约 40-65 秒延迟，完全不可接受
- **建议**：GeneralAgent 的 LLM 调用应仅用于玩家控制势力（`isHumanSlot=true`），AI势力全部使用 GeneralUtilityAI 规则决策（当前 `skipLLM=true` 已是正确实现）
